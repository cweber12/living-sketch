# Implementation Plan: Per-Module Agent Context Files

### Purpose

When an AI agent works on a specific module (Sketch, Extract, or Console), it currently must either load the full 143-file `docs/architecture.md` or navigate the codebase blind. The full architecture doc contains significant noise for single-module work — Sketch agents don't need Console's rendering pipeline, and Console agents don't need Sketch's SVG toolbar constants. The goal is to produce focused, auto-generated context files scoped to each module, automatically surfaced to agents via VS Code Copilot's `applyTo` instruction loading.

### Intended Result

After implementation, when an agent opens any file in `app/extract/**` or `components/extract/**`, VS Code Copilot automatically loads `.github/instructions/extract-module.instructions.md`, which tells the agent: what this module does, what files it owns, what libraries it depends on exclusively, what it shares with other modules, and which API routes it uses — all derived from the live import graph with no manual upkeep.

The same applies to Console. Sketch already has a hand-written `sketch-page.instructions.md` — that file is preserved and extended only with a reference to the generated context file.

---

### Codebase Context (for reviewers with no workspace access)

**Project**: `living-sketch` — Next.js 16 (App Router), React 19, TypeScript strict mode, Tailwind CSS v4. A pose-driven animation app. Three user-facing pages (Sketch, Extract, Console) plus authentication routes.

**Three core modules:**

- **Sketch** — Users draw SVG artwork onto 14 body-part canvases. Saves SVGs to Supabase storage. Owns: `app/sketch/`, `components/sketch/`, `hooks/use-sketch-canvas-rig.ts`.
- **Extract** — Webcam-based pose detection using MediaPipe. Saves detected landmark data to Supabase storage. Owns: `app/extract/`, `components/extract/`, `hooks/use-pose-detection.ts`.
- **Console** — Fetches saved landmarks and SVGs, runs 3D rendering pipeline, plays back pose-driven animation. Owns: `app/console/`, `components/console/`, `hooks/use-cache-svgs.ts`.

Modules communicate **only** through HTTP calls to `app/api/storage/` route handlers backed by Supabase. There is no shared React state between modules.

**Architecture generator**: `scripts/gen-arch.mjs` — a Node.js ES module script (~500 lines) using the `typescript` package (already a devDep) to parse import/export ASTs across 143 source files. Outputs `docs/architecture.md`. Run via `npm run arch`. The script already computes:

- `depMap`: `Map<relPath, { internal: Set<relPath>, external: Set<pkgName>, exports: string[], role: string }>`
- `reverseMap`: `Map<relPath, Set<relPath>>` — which files import a given file
- `moduleImports`: `{ sketch: Set<dep>, extract: Set<dep>, console: Set<dep> }` — all transitive deps used by each module
- `moduleExclusive[mod]`: array of deps used by exactly one module
- `sharedDeps`: array of `{ dep, usedBy, kind }` where kind is `functional | state | supabase | api | ui`
- `MODULE_DEFS`: config constant defining each module's `prefixes` and `ownFiles`
- `classifyFile(rel)`: returns a role string (Page, Hook, Store, Util, Types, Constant, Canvas, Toolbar, Icon, etc.)

**Existing instruction file**: `.github/instructions/sketch-page.instructions.md` — hand-written, ~120 lines. Uses YAML frontmatter:

```yaml
---
applyTo: 'app/sketch/**,components/sketch/**,hooks/use-sketch-canvas-rig.ts'
---
```

VS Code Copilot automatically injects this file's content into agent context whenever the agent opens a file matching the `applyTo` glob. This file contains detailed behavioral rules, toolbar architecture notes, session state keys, etc. It must **not** be overwritten by the generator.

No equivalent instruction files exist yet for Extract or Console.

### Changes Required

#### 1. Extend `scripts/gen-arch.mjs` — add per-module output

Add a `writeModuleDoc(mod)` function that runs after the existing dependency analysis (which already has all the data needed) and writes `docs/modules/{mod}.md`.

Each module doc is structured as an **agent task map** — not just a filtered file index. The goal is that an agent can read the first two sections and already understand what the module is, what it owns, and what it must not break, before reading any source file.

**Generated-file header (top of every module doc)**
Each file begins with:

```md
> Auto-generated by `npm run arch`. Do not edit manually.
> If this file conflicts with source code, trust the source code and regenerate.
```

**Section 0 — Start Here**
A short (4–6 line) summary that directly answers the questions an agent needs before touching any code:

- What does this module do?
- What files does it own?
- What data does it produce or consume?
- What storage/API routes does it use?
- What dependencies are shared and risky to edit?
- What files are currently module-exclusive (safer to change)?

This section must appear before any file index so agents do not need to scan a long table to orient themselves.

**Section 1 — Module Header**
One-paragraph description from `MODULE_DEFS[mod].desc`. Lists owned paths (`prefixes` + `ownFiles`).

**Section 2 — Entry Points**
Small table: Page file, Layout file, primary Hook file — with their roles and top-level exports. These are deterministic from `MODULE_DEFS` and `depMap`.

**Section 3 — Owned File Index**
All files whose `getModuleOwnership(rel)` returns this module, sub-grouped by `groupKey(rel)`. Format matches the main architecture doc's File Index but filtered to this module only. Each entry: path, role, exports (truncated to 6), key internal imports. File paths and export lists sorted alphabetically.

**Section 4 — Module-Exclusive Dependencies**
The `moduleExclusive[mod]` list, already computed. For each dep: path, role, exports, and which module file(s) import it.

Wording: These files are currently only used by this module based on the import graph. They are safer to modify than shared dependencies, but public exports and data/storage contracts should still be preserved unless intentionally changed. A file can be exclusive today but still represent an important stable contract.

**Section 5 — Shared Dependencies (handle with care)**
From `sharedDeps`, filtered to entries where `usedBy.includes(mod)`. This is the most important section for preventing cross-module breakage. For each shared dep show:

- File path and role
- Dependency kind: `functional`, `state`, `supabase`, `api`, or `ui`
- All modules that use it
- Which files within **this module** import it (derived from `depMap` reverse lookup filtered to module-owned files)
- A one-line note explaining the risk of changing it

Split into subsections: Functional, Supabase, Shared UI. Sorted by path within each subsection.

**Section 6 — Storage Bridge**
Derived from the import graph: identify which `app/api/storage/` route files are imported or fetched by module-owned files. For each route, show:

- Route path (e.g. `POST /api/storage/landmarks`)
- HTTP method (derived from the route file's exported handler names: `GET`, `POST`, etc., parsed from `extractExports`)
- Whether this module reads or writes
- Data type involved (e.g. landmark JSON, SVG file)
- Which module produces the data and which consumes it

Fall back to hand-maintained descriptions only for routes whose HTTP method cannot be inferred from exports. Note which entries are derived vs. hand-maintained.

**Section 7 — Packages Used**
`external` packages from `depMap`, filtered to files owned by this module. Deduplicated. Sorted alphabetically.

**Section 8 — Boundary Warnings**
The generator flags suspicious architectural conditions and emits this section (omitted if no warnings exist):

- A module-owned file imports another module's owned file (cross-module coupling in the import graph)
- A file classified as module-exclusive is also found in another module's `moduleImports` (misclassification risk)
- An expected entry point (page, layout, primary hook from `MODULE_DEFS`) is missing from `depMap`
- `lib/3d/**` files appear in a module other than Console's import set (ownership drift)
- Any file used by the module that `classifyFile` returns `'Module'` (unclassified — may need a new role)

Warnings are surfaced in `docs/modules/{mod}.md` and also printed to stdout during `npm run arch` so they are visible in CI.

**Determinism requirement**
A second run of `npm run arch` with no source changes must produce identical output. All collections that feed into generated text must be sorted before use: file paths, import lists, export lists, external package lists, shared dep lists, reverse-importer lists, and module section ordering.

The existing `writeArchDoc()` function continues to write `docs/architecture.md` unchanged. The new `writeModuleDoc(mod)` function is called three additional times at the end of the script.

Update the `console.log` at the end to report all four output files and any boundary warnings.

`docs/modules/` directory must be created if it doesn't exist (`fs.mkdirSync(path.join(ROOT, 'docs', 'modules'), { recursive: true })`).

### Design Decisions and Tradeoffs

**Why separate generated docs from instruction files?**
`sketch-page.instructions.md` contains ~120 lines of hand-written behavioral rules (toolbar state invariants, session key names, canvas architecture). Auto-generating it would overwrite that on every `npm run arch` run. Keeping generated content in `docs/modules/` and hand-written rules in `.github/instructions/` lets both evolve independently.

**Why not embed the generated content inline in instruction files?**
Embedding would require parsing and updating the instruction files on each run — complex to implement without risking data loss. A reference link is simpler and equally effective: agents can follow the link and load `docs/modules/extract.md` when needed.

**Why include `lib/3d/**`in Console's`applyTo`?**
The architecture analysis confirms `lib/3d/`is Console-exclusive based on the current import graph. An agent editing a 3D pipeline file needs Console module context. The Console instruction file explicitly documents this condition and instructs maintainers to update the`applyTo`if ownership drifts. The boundary warnings section of`docs/modules/console.md`will flag it if`lib/3d/` is imported by another module.

**What about `lib/types.ts`, `lib/utils/frame-filter.ts`, `lib/utils/landmark-smoother.ts`?**
These are shared between Extract and Console. They are excluded from module `applyTo` patterns because their correct editing context is the Shared Dependencies section in both `extract.md` and `console.md`, plus the Module Relationships section in the full `architecture.md`. An agent editing `lib/types.ts` should load `architecture.md` rather than a single module doc.

**Why add boundary warnings?**
The generator is the authoritative source of architectural truth for this codebase. Boundary warnings make violations visible at generation time — in CI output and in the generated doc — rather than leaving them as silent structural drift. This turns the generator from a passive documentation tool into an active architecture linter.

**Why derive the Storage Bridge from exports rather than hardcoding?**
Hardcoded route descriptions go stale as API routes are added or changed. Deriving HTTP methods from exported handler names (`GET`, `POST`, etc. parsed by `extractExports`) keeps route descriptions accurate without manual maintenance. Route descriptions that cannot be inferred are flagged as hand-maintained in the output.

**Why require deterministic output?**
If a second `npm run arch` run with no source changes produces a diff, agents or CI that re-run the script will create noisy commits. Sorting all collections before rendering ensures the output is stable and meaningful diffs reflect real codebase changes.

### Constraints

This is a documentation and context-generation refactor only. The implementing agent must not touch:

- Application behavior (hooks, UI logic, rendering, webcam, pose detection)
- Storage or API implementation (`app/api/`)
- Supabase client configuration
- Any test files
- `package.json`, `tsconfig.json`, or build configuration

The only application-adjacent change is the additive reference line inserted into `.github/instructions/sketch-page.instructions.md`.
